# Valkey Authentication and TLS Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add authentication (ACL) and TLS encryption to the Valkey connection in the SearXNG Helm chart, supporting internal (subchart) and external Valkey servers.

**Architecture:** Unified `valkey` config block with `mode` (internal/external), `auth`, and `tls` sections. Password injected via `SEARXNG_REDIS_URL` env var using Kubernetes dependent env var syntax `$(VALKEY_PASSWORD)`. TLS certs generated by cert-manager for internal mode, user-provided for external. Backward compatible — defaults preserve current behavior.

**Tech Stack:** Helm 3, cert-manager v1, Valkey subchart v0.9.3 (ACL auth + TLS)

**Design doc:** `docs/plans/2026-03-07-valkey-auth-tls-design.md`

### Task 8: Wire Subchart Passthrough Values

**Files:**
- Modify: `charts/searxng/values.yaml` (valkey subchart passthrough for auth and TLS)

The Valkey subchart needs to receive `auth` and `tls` config from our parent values. Helm automatically passes values under the `valkey:` key to the subchart. We need to ensure the subchart gets the right auth and TLS settings when enabled.

**The problem:** Our new `valkey.auth` and `valkey.tls` blocks in values.yaml will pass through to the Valkey subchart automatically. However, the subchart's `auth` schema expects `aclUsers` and `usersExistingSecret`, which our simplified API doesn't expose directly.

**The solution:** We need an intermediate template that constructs the subchart values. Since Helm doesn't support dynamic subchart value overrides in templates, we use the subchart's `auth` and `tls` fields directly in our values — our `valkey.auth` and `valkey.tls` blocks happen to nest under the same `valkey:` key that passes to the subchart.

**Step 1: Add subchart auth and TLS passthrough fields to the valkey block**

In `values.yaml`, after the `tls` block and before the `# --- Subchart passthrough values ---` comment, the existing structure already passes through. However, we need to handle the mapping.

The Valkey subchart expects:
- `auth.enabled` — we have this
- `auth.usersExistingSecret` — we need to set this dynamically
- `auth.aclUsers` — we need to define the default user
- `tls.enabled` — we have this
- `tls.existingSecret` — we need to set this dynamically

Since Helm subchart values can't be templated in `values.yaml`, we need to handle this differently. The subchart values are static — we must use the deployment-time mechanism.

**Revised approach:** We already have our parent chart's `valkey.auth` and `valkey.tls` passing through to the subchart. The fields we added (`password`, `existingSecret` as our key, `key`, `certManager`) are not subchart fields — the subchart will ignore them. But `auth.enabled` and `tls.enabled` will pass through correctly.

For the dynamic subchart fields (`usersExistingSecret`, `aclUsers`, `tls.existingSecret`), we need to document that users deploying with auth must set them explicitly in their values override.

**Actually, let's handle this properly:** Add the subchart-specific auth fields as defaults in `values.yaml` that work when auth is enabled.

Add these fields to the `valkey:` block in `values.yaml`, after the `tls` section and before `podSecurityContext`:

```yaml
  # --- Valkey subchart auth passthrough (internal mode only) ---
  # These map directly to the Valkey subchart's auth configuration.
  # When auth.enabled is true, the subchart uses these to configure ACL.
  # See: https://github.com/valkey-io/valkey-helm
  #
  # NOTE: usersExistingSecret is set dynamically — override in your
  # deployment values when using an existingSecret for the password.
  # For chart-generated passwords, the default ACL user config works
  # as-is with the generated Secret name.
  aclUsers:
    default:
      permissions: "~* &* +@all"
      passwordKey: "valkey-password"
```

Wait — this won't work cleanly because the subchart expects `auth.aclUsers`, not `aclUsers` at the top level of the `valkey:` block. It IS under `valkey.auth.aclUsers` in the subchart's schema, and our `valkey.auth` block will pass through.

Let me re-examine the subchart values structure:

The subchart expects under its root:
```yaml
auth:
  enabled: false
  usersExistingSecret: ""
  aclUsers: {}
tls:
  enabled: false
  existingSecret: ""
```

Our parent chart's `valkey:` block maps to the subchart root. So:
- `valkey.auth.enabled` → subchart `auth.enabled` ✓
- `valkey.auth.existingSecret` → subchart `auth.existingSecret` — BUT subchart has no `existingSecret`, it has `usersExistingSecret`
- `valkey.tls.enabled` → subchart `tls.enabled` ✓
- `valkey.tls.existingSecret` → subchart `tls.existingSecret` ✓

The naming conflict is `auth.existingSecret` (our field) vs `auth.usersExistingSecret` (subchart field). Our field name doesn't collide — the subchart ignores unknown fields.

    aclUsers:

        permissions: "~* &* +@all"

## Valkey Configuration

The chart supports three Valkey deployment modes:

### Internal Valkey (Default)

The chart deploys a bundled Valkey instance via subchart. By default, TLS is
enabled via a self-signed cert-manager Certificate.

```yaml
valkey:
  mode: "internal"
  tls:
    enabled: true  # Default — auto-generates certs via cert-manager
```

### Internal Valkey with Authentication

Enable password authentication for the bundled Valkey instance:

```yaml
valkey:
  mode: "internal"
  auth:
    enabled: true
    # usersExistingSecret must match the generated Secret name:
    # <release-name>-searxng-valkey-auth
    usersExistingSecret: "searxng-valkey-auth"
  tls:
    enabled: true
```

The chart generates a random password stored in a Kubernetes Secret and
configures both Valkey (ACL) and SearXNG (`SEARXNG_REDIS_URL` env var)
to use it. The password is preserved across `helm upgrade`.

### External Valkey

Connect to an existing Valkey/Redis server:

```yaml
valkey:
  mode: "external"
  external:
    host: "redis.example.com"
    port: 6379
    db: 0
  auth:
    enabled: true
    existingSecret: "my-redis-password"
    key: "password"
  tls:
    enabled: true
    existingSecret: "my-redis-tls"
```

When using external mode, the bundled Valkey subchart is not deployed.
You must provide:
- An existing Secret with the password (referenced by `auth.existingSecret`)
- An existing Secret with TLS certificates (`ca.crt`, `tls.crt`, `tls.key`)

### Vault Secrets Operator (VSO) Example

For external Valkey with VSO-managed credentials:

```yaml
valkey:
  mode: "external"
  external:
    host: "redis.prod.internal"
    port: 6379
    db: 0
  auth:
    enabled: true
    existingSecret: "valkey-auth-secret"
    key: "password"
  tls:
    enabled: true
    existingSecret: "valkey-tls-certs"
```

Create the VaultStaticSecret to sync the password:

```yaml
apiVersion: secrets.hashicorp.com/v1beta1
kind: VaultStaticSecret
metadata:
  name: valkey-auth-secret
spec:
  type: kv-v2
  mount: StaticSecrets
  path: valkey
  destination:
    name: valkey-auth-secret
    create: true
  refreshAfter: 60s
  vaultAuthRef: vault/vault-auth
```
