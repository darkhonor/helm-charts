# Valkey Authentication and TLS Design

> **Finding:** SRXNG-VULN-005 — Unauthenticated, unencrypted Valkey connection
> **Issue:** darkhonor/helm-charts#64
> **Priority:** CAT II (Medium)

**Goal:** Add authentication and TLS encryption to the Valkey connection in the
SearXNG Helm chart, supporting both the bundled subchart and external Valkey
servers.

**Architecture:** Unified `valkey` config block with `mode` (internal/external),
`auth`, and `tls` sections. Password injected via `SEARXNG_REDIS_URL` env var
using Kubernetes dependent environment variable syntax. TLS certs generated by
cert-manager for internal mode, user-provided for external.

**Tech Stack:** Helm, cert-manager, Valkey subchart v0.9.3 ACL auth

---

## Values Schema

```yaml
valkey:
  # "internal" deploys the bundled subchart
  # "external" connects to an existing Valkey server
  mode: "internal"

  # External Valkey connection (only when mode: external)
  external:
    host: ""
    port: 6379
    db: 0

  # Authentication (applies to both modes)
  auth:
    enabled: false
    # Leave empty to auto-generate (internal mode only)
    password: ""
    # Use an existing Secret instead of chart-generated
    existingSecret: ""
    # Key within the Secret containing the password
    key: "valkey-password"

  # TLS encryption (applies to both modes)
  tls:
    enabled: true
    # For external mode: user provides a Secret with ca.crt, tls.crt, tls.key
    existingSecret: ""
    # For internal mode: chart creates a cert-manager Certificate
    certManager:
      issuerRef:
        name: ""   # Empty = chart creates a self-signed Issuer
        kind: "Issuer"
```

## Scenarios

| # | Mode | Auth | TLS | Behavior |
|---|------|------|-----|----------|
| 1 | internal | off | on | Subchart deployed, TLS via cert-manager, no password |
| 2 | internal | on | on | Subchart with ACL user, chart-generated password, TLS via cert-manager |
| 3 | external | on | on | No subchart, user provides host + existingSecret for password + TLS certs |

## Secret Management

### Chart-Generated Password

When `auth.enabled: true` and no `existingSecret`, the chart creates a Secret:

```yaml
# templates/valkey-auth-secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: {{ fullname }}-valkey-auth
type: Opaque
data:
  valkey-password: {{ lookup or randAlphaNum 32 | b64enc }}
```

Uses the `lookup` function to preserve the password across `helm upgrade`,
same pattern as `searxng-secret`.

### Password Flow

1. Chart creates the Secret (or user provides `existingSecret`)
2. Valkey subchart receives `auth.usersExistingSecret` pointing to the Secret
3. Valkey init script reads the password and creates the ACL entry
4. SearXNG deployment reads the same Secret via `secretKeyRef` for the env var

### Valkey Subchart Passthrough (internal + auth)

```yaml
valkey:
  auth:
    enabled: true
    usersExistingSecret: "{{ fullname }}-valkey-auth"
    aclUsers:
      default:
        permissions: "~* &* +@all"
        passwordKey: "valkey-password"
```

Single `default` user with full permissions, password from the shared Secret.

## TLS Architecture

### Internal Mode — cert-manager

Three resources created:

1. **Self-signed Issuer** (when `tls.certManager.issuerRef.name` is empty):

```yaml
# templates/valkey-tls-issuer.yaml
apiVersion: cert-manager.io/v1
kind: Issuer
metadata:
  name: {{ fullname }}-valkey-selfsigned
spec:
  selfSigned: {}
```

2. **Certificate** referencing the Issuer:

```yaml
# templates/valkey-tls-certificate.yaml
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: {{ fullname }}-valkey-tls
spec:
  secretName: {{ fullname }}-valkey-tls
  issuerRef:
    name: {{ issuerRef.name | default fullname-valkey-selfsigned }}
    kind: {{ issuerRef.kind }}
  dnsNames:
    - {{ fullname }}-valkey
    - {{ fullname }}-valkey.{{ namespace }}.svc
    - {{ fullname }}-valkey.{{ namespace }}.svc.cluster.local
  duration: 8760h    # 1 year
  renewBefore: 720h  # 30 days
```

3. cert-manager creates the **Secret** with `ca.crt`, `tls.crt`, `tls.key`

### Passthrough to Valkey Subchart

```yaml
valkey:
  tls:
    enabled: true
    existingSecret: "{{ fullname }}-valkey-tls"
```

### External Mode

User provides `valkey.tls.existingSecret` with their own certs. No Issuer or
Certificate created. SearXNG mounts the CA from that Secret.

## Deployment Template Changes

### Environment Variables

```yaml
env:
  # Conditional: only when auth enabled
  - name: VALKEY_PASSWORD
    valueFrom:
      secretKeyRef:
        name: {{ auth secret name }}
        key: {{ auth key }}
  # Always present: built from values
  - name: SEARXNG_REDIS_URL
    value: "{{ scheme }}://:$(VALKEY_PASSWORD)@{{ host }}:{{ port }}/{{ db }}"
```

URL components resolved from values:

| Component | Internal | External |
|-----------|----------|----------|
| scheme | `rediss` (TLS) or `redis` | `rediss` (TLS) or `redis` |
| host | `{{ fullname }}-valkey.{{ ns }}.svc.cluster.local` | `valkey.external.host` |
| port | `6379` | `valkey.external.port` |
| db | `0` | `valkey.external.db` |

Kubernetes resolves `$(VALKEY_PASSWORD)` at container start from the
previously defined env var (dependent environment variable syntax).

### Volume Mount — CA Certificate

```yaml
volumeMounts:
  - name: valkey-tls-ca
    mountPath: /etc/ssl/valkey/ca.crt
    subPath: ca.crt
    readOnly: true
volumes:
  - name: valkey-tls-ca
    secret:
      secretName: {{ tls secret name }}
      items:
        - key: ca.crt
          path: ca.crt
```

### Settings Cleanup

Remove hardcoded `valkey.url` from default `config.settings.data` in
`values.yaml`. The `SEARXNG_REDIS_URL` env var overrides it at runtime.

## Subchart Conditional Deployment

```gotemplate
{{- define "searxng.valkeyEnabled" -}}
{{- eq .Values.valkey.mode "internal" -}}
{{- end -}}
```

- `mode: internal` → subchart deployed (`valkey.enabled: true`)
- `mode: external` → subchart disabled (`valkey.enabled: false`)

The existing `condition: valkey.enabled` in `Chart.yaml` handles this.

## Helper Templates

New helpers in `_helpers.tpl`:

- `searxng.valkeyEnabled` — boolean, is subchart deployed
- `searxng.valkeyHost` — resolves host from mode
- `searxng.valkeyPort` — resolves port from mode
- `searxng.valkeyDB` — resolves db from mode
- `searxng.valkeyScheme` — `rediss` or `redis` based on TLS
- `searxng.valkeyAuthSecretName` — resolves to existingSecret or generated name
- `searxng.valkeyTlsSecretName` — resolves to existingSecret or generated name
- `searxng.redisURL` — assembles the full URL

## NIST Controls Addressed

- **SC-8** (Transmission Confidentiality) — TLS on all Valkey connections
- **SC-28** (Protection at Rest) — password in Secret, not ConfigMap
- **IA-5** (Authenticator Management) — ACL-based auth with password from Secret
- **AC-6** (Least Privilege) — single default user with defined permissions

## Backward Compatibility

Default values (`auth.enabled: false`, `mode: internal`) preserve current
behavior. Existing deployments upgrade without changes.
